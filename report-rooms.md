# Design Review Report

**Document Reviewed:** Game Room / Code-Based Join Flow  
**Review Date:** 2026-06-02  
**Complexity Rating:** 🟡 Medium  
**Reviewer Persona:** Senior Staff Engineer / Architect

---

## Executive Summary

The proposal is sound and well-scoped. Adding a 3-letter game code is a clean solution to the idle-player pollution problem — it gives the admin a hard reset lever (kill game, new code) and prevents stale socket connections from old sessions bleeding into new games. The single-active-game constraint keeps the state machine simple. This is approved with minor conditions around the DB schema and kill-game broadcast.

**Verdict:** ⚠️ Approved with Conditions

---

## Strengths

- Keeping one active game at a time avoids multi-tenancy complexity while still solving the isolation problem.
- Storing the code in the DB is the right call — it survives the admin refreshing the home page and gives a clean invalidation hook.
- Admin-only home page (password gate before create) means the code can't be generated by a random guest.
- The "Game closed" broadcast is the correct UX — players get immediate feedback rather than silently losing connection.

---

## Issues Identified

### 🔴 Critical (must fix before proceeding)
- **Socket.IO rooms must be scoped to the game code.** Currently `io.emit(...)` broadcasts to every connected socket. With a code-based system, all broadcasts must be `io.to(code).emit(...)` so players from a killed game don't receive events from a new game. Every `io.emit` and `socket.emit` in the game engine needs auditing.

### 🟡 Medium (should address before launch)
- **Game code collision**: 3 letters = 17,576 combinations. With one active game at a time this is fine, but the generator should retry on collision with whatever is in the DB.
- **Finished vs killed code UX**: both show "Invalid game code" — consider showing "Game finished" vs "Game was closed" for better player experience. Low effort, high clarity.
- **Home page after kill**: when admin kills a game, should the home page reset so they can create a new one immediately? Currently unclear — the admin is on `/game/{code}` when they kill it.

### 🟢 Low (nice to have)
- Show the game code prominently on the `/game/{code}` lobby screen (not just the home page) so the admin can re-display it on the TV if needed.
- QR code pointing to `/game/{code}` on the lobby screen would make player joining seamless.

## Decisions Confirmed

- **One active game at a time** — no multi-room complexity.
- **Code stored in DB** — survives admin page refresh; clean invalidation on kill/finish.
- **Home page is admin-only** — password required before "Create Game" is available.
- **Server restart = code gone** — consistent with the existing no-crash-recovery decision.
- **Static error page** for invalid/killed/finished codes — no redirect.
- **Kill game broadcasts "Game closed"** to all players in the Socket.IO room.

## Recommendation

Greenlight to implement. The key implementation steps in order:
1. Add a `games` table to the DB schema (`code TEXT PRIMARY KEY`, `status TEXT`, `created_at`, `finished_at`).
2. Update home page: password gate → "Create Game" button → generate + store code → show code big → nav button to `/game/{code}`.
3. Move `/game/page.tsx` → `/game/[code]/page.tsx`; validate code on load, show error page if invalid.
4. Scope all game engine state and Socket.IO emits to the game code (join room on connect, use `io.to(code).emit` everywhere).
5. Add `admin:killGame` event → set game status to killed in DB → broadcast "game:closed" to room → redirect admin to `/`.
